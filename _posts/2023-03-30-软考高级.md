## 5 信息系统治理

### 审计风险
1. 固有风险：客观存在的风险。it活动不存在相关控制的话，会导致重大错误
   1. 分类
   2. 特点
      1. it活动本身具有的，审计人员职能评估，无法控制
      2. 固有风险的是主观的、复杂的，不同的it活动的固有风险水平不同
2. 控制风险：内部控制体系不能及时检查或预防的重大错误
   1. 特点 ：
      1. 与内部制度执行有关，与审计无关，审计人员只能评估水平，不能实施影响和控制
      2. 需要传统内部控制的思想和计算机系统的管理知识，较复杂而难以准确计量
3. 检查风险：审计程序未发现的错误
   1. 原因
      1. it审计不规范
      2. 审计人员本身或技术原因影响
4. 总体审计风险：各类审计风险的总和

### 审计标准
1. 信息系统审计准则ISACA
2. 内部控制-整体框架COSO
3. 萨班斯法案SOX
4. 信息及相关技术控制目标COBIT

### 审计技术
1. 风险评估技术
   1. 风险识别技术：德尔菲法、头脑风暴法、检查表法、SWOT
   2. 风险分析技术：定量分析和定性分析
   3. 风险评价技术：对风险程度进行划分，包括单个风险评价和总体风险评价
   4. 风险应对技术：云计算、冗余、冗余链路、冗余资源、系统弹性伸缩、两地三中心灾备，业务熔断限流

### 审计证据

### 审计底稿
审计员对制定的审计计划，实施的审计程序，获取的相关的审计证据，以及得出的审计结论的记录
1. 分类
   1. 综合类的底稿：审计计划阶段、和审计报告阶段，为规划、控制和总结整个审计工作并发表审计所见形成的审计工作底稿
   2. 业务类的底稿：实施阶段为执行审计程序形成的底稿
   3. 备查类的底稿：仅具有被查类的底稿

### 审计的四个阶段
1. 审计准备阶段：从计划开始到发出审计通知书为止
   1. 明确审计任务
   2. 组件审计项目组
   3. 搜集相关信息
   4. 制定审计计划
2. 审计实施阶段
   1. 深入调查并调整审计计划
   2. 了解初步评估it内部
3. 审计终结阶段：整理底稿、编写报告、做出总结
   1. 整理底稿
   2. 整理证据
   3. 评估相关目标的实现
   4. 报告审计发现
   5. 沟通审计结果
   6. 出具审计报告
   7. 归档管理
4. 后续审计阶段：后续检查是否采取纠正措施

## 8 项目管理理论

### 项目基础
1. 项目的定义：项目是为了创造独特的产品、服务或成果而进行的临时性工作。
   1. 独特的产品、服务或成果：可以是有形的也可以是无形的。是想项目目标可能会产生多个可交付成果。某些项目的可交付成果可以存在相同的元素，但不会改变项目本质上的独特性。项目可以在组织上的任何层级开展。
   2. **临时性工作**：有明确的起点和终点。可交付成果可能会在项目终止后仍然存在。
   3. 项目的驱动变更
2. 项目管理不善可能导致
   1. 超过时限
   2. 成本超支
   3. 质量低劣
   4. 返工
   5. 范围失控
   6. 声誉受损
   7. 干系人不满意
   8. 无法达成目标

### 项目、项目、项目组合和运营管理之间的关系
1. 组织的角度：
   1. 项目和项目集管理的重点在于**正确的方式开展**项目集和项目
   2. 项目组合管理重点在**开展正确**的项目集和项目
2. 项目集管理：管理项目之间的依赖关系
3. 项目组合管理：项目集之间的项目集或项目不存在依赖关系。
4. 运营管理：不属于项目管理范畴
5. 组织级项目管理和战略：项目组合、项目集和项目都需要符合组织战略。

### 项目内外部的运行环境
1. 组织过程资产
   过往经验或项目经理可以改变的文件
   1. 过程资产：工具、方法论
   2. 治理文件：项目的政策和流程
   3. 数据资产：以前项目的数据库
   4. 知识资产：项目成员、主题专家
   5. 安保和安全：数据访问、数据保护
2. 组织内部的事业环境因素（公司层面）
   1. 组织文化：愿景、使命、价值观
   2. 设施和资源的物理分布：工作地点
   3. 基础设施：设施、设备、it硬件
   4. 信息技术软件
   5. 资源可用性
3. 组织外部的事业环境因素
   1. 市场条件
   2. 社会和文化影响因素
   3. 监管因素
   4. 商业数据库
   5. 学术研究
   6. 行业标准
   7. 财务考虑因素
   8. 物理环境因素

### 组织系统
1. 治理框架
2. 管理要素
3. **组织结构类型** ：

   1. 组织类型中项目经理的权力大小
      1. 职能型组织：职能经理作为项目经理，不适合大型跨部门项目，中间协调员没有权利，有利于专业技能的培养
      2. 弱矩矩型组织：没有专职的项目经理由职员担任pm作为协调员，权力有限
      3. 平衡矩阵组织：项目经理开始全职化，权力受到职能经理限制
      4. 强矩阵组织：有独立项目部门，没有完全独立的项目资源，pm与只能经理权力相同，但资源需要谈判
      5. 项目型组织：项目经理拥有很大自主性和职权，有完全的资源调配权力，项目经理权力大，对资源又控制权，资源使用效率低下，缺乏归宿感，不利于员工技能培养急职业发展
      6. 复合型组织：大项目由项目经理管理，小项目有职员兼职项目经理进行管理

      2. 控制性：要求项目服从文档制度
      3. 指令型：指导项目进行，pm负责细节执行

1. 项目管理：帮助达成项目目标
2. 战略和商务：有助于提高绩效并取得更好的业务成果
3. 领导力：帮助组织达成业务目标
   1. 人际交往
   2. 领导者品质和技能
      1. 远见
      2. 乐观
      3. 乐于合作
      4. 管理关系和冲突：建立信任、解决顾虑、寻求共识、运用说服、协商、妥协和解决冲突的技能、发展和培养个人社会关系和职场人脉，用长远的眼光把人际关系堪称与项目同样重要，持续发簪和运用职业敏锐性。
      5. 花大量实践狗托那个，管理期望，诚恳的接受反馈，提出建设性意见，询问和倾听
      6. 尊重他人（保持独立自主），谦恭有礼、友善待人、诚实可信、忠诚可靠、遵守职业道德
      7. 诚信正值和文化敏感性，果断、勇敢、能够解决问题
      8. 适当称赞他人
      9. 终生学习
      10. 管事重要的事情
      11. 以整体和系统的角度看待项目
      12. 运用批判性思维
      13. 创建高效团队
   3. 政策和权力：项目经理需要在组织政策、协议和程序许可的范围内主动需求所需的权力和职权，而不是坐等组织授权。
   4. 领导力和管理：同时采用领导力和管理两种方式，找到平衡点。
      1.  管理：指挥一个人执行一系列一致的预期行为。
      2.  领导力：带领一些人从一个位置到另一个位置。  
      
      **领导力风格**
      1. 放任型：允许团队自主决策和设定目标
      2. 交易型：依据目标、反馈和成就给予奖励
      3. 服务型：处处为他人着想
      4. 变革型：理想化特质和行为、鼓舞性激励、促进创新和创造
      5. 魅力型：说服力强
      6. 交互性：交易型、变革型和魅力型

#### 项目生命周期和项目阶段
1. **项目生命周期适用于任何项目。所有项目都包含启动项目、组织于准备、执行项目工作和结束项目。**

   项目生命周期特征：
   1. 成本和人力投入开始比较低，项目执行期间最高，并在项目快要结束时迅速回落
   2. 风险与不确定性开始最高，后面最低。开始变更代价最小，后面变更代价最大。
2. **项目生命周期类型** ：**不同阶段可以采用不同生命周期**
   1. 预测型的生命周期：瀑布型：已经充分了解并明确需求的项目。
   2. 迭代型的生命周期：修改优化新增功能
   3. 增量型生命周期：增加功能
   4. 适应型生命周期：敏捷性：变更驱动型：特点是基于初始需求制定一套高层级计划，在逐渐把需求细化到合适特定规划周期的详细程度
   5. 混合型生命周期：预测型和适应型混合

### 制定项目章程
是编写一份正式批准项目并授权项目经理在项目活动中使用组织资源的文件的过程。

作用是：
1. 明确项目与组织战略目标之间的直接关系
2. 确立项目的正式地位
3. 展示组织对项目的承诺

本过程只开展一次

- 项目章程在项目执行和项目需求之间建立了联系。
- **项目章程不能当作合同，通常需要正式的合同来达成合作协议。项目章程用于建立组织内部的合作关系。**  
- **在规划开始前就要任命项目经理，最好在制定项目章程时就任命。**  
- **项目章程可由发起人编制，也可由项目经理与发起机构合作编制。**  
- **项目章程一旦被批准，就标志着项目正式启动**  
- **项目由项目以外的机构来启动，例如发起人，项目集或项目管理办公室，项目组合治理委员会主席及其授权代表**
- **项目启动人应该有一定的职权，能为项目获取资源**

### 创建WBS
作用是是吧项目可交付成果和项目工作分解成较小的、更易管理的组件的过程。
WBS最底层的组成部分称为**工作包**。工作包是可交付成果，不是活动本身。
**仅开展一次或仅在项目的预定义点开展**
#### 输入

1. 项目管理计划
2. 项目文件（需求文件，项目范围说明书）
3. 事业环境因素
4. 组织过程资产

#### 技术

1. 专家判断
2. **分解**：将项范围和项目可交付成果逐步划分为更小、更便于管理的组成部分的技术。工作分解得越细致，对工作的规划、管理和控制就越有力。但是，过细的分解会造成管理努力的无效耗费资源使用效率低下、工作实施效率降低，同时造成 WBS 各层级的数据汇总困难。
   1. 分解活动/步骤
      1. 识别和分析可交付成果和相关工作（看有什么需要分解）
      2. 确定wbs结构和编排方法（确定分解方法）
      3. 自上而下逐层细化分解（开发分解）
      4. 为WBS组成部分制定（编码标识分解内容）
      5. 核实可交付成果分解程度是否恰当（检查分解内容）
    2. WBS结构：
       1. 项目生命周期作为第二层，，把产品和项目可交付成果放在第三层
       2. 可交付成果作为第二层
       3. 纳入由项目团队意外的组织的各种较低层次组件（外包工作）
    3. 注意事项/原则
       1. WBS 必须是面向可交付成果的
       2. WBS必须符合项目的范围：100%原则，在WBS中所有下级元素的和必须代表上一级元素
       3. WBS的底层应该支持计划和控制，**WBS是项目管理计划和项目范围之间的桥梁**
       4. WBS的元素必须有人负责，且只有一个人负责。
       5. WBS应该控制在4~6层。一个工作单元只能从属于某个上层单元，避免交叉从属。
       6. WBS应包括项目管理工作和外包工作
       7. WBS的编制需要**所有（主要）**项目干系人参与
       8. WBS并非一成不变的。完成之后也可修改。

#### 输出

1. 范围基准：是项目管理计划的组成部分
   1. 项目范围说明书
   2. WBS
   3. 工作包：WBS 的最低层是带有独特标识号的工作包。是已经确定的工作。**控制账号**是一个控制点，包含**两个以上的工作包**，一个包只能属于一个控制账号。
   4. 规划包：规划包是一种低千控制账户而高千工作包的工作分解结构组件。是待确定的工作。**控制账号**可以有**一个或多个规划包**。
   5. WBS字典

2. 项目文件（更新）：假设日志、需求文件

### 确认范围
作用是正式验收已完成的项目可交付成果的过程。**在整个项目期间定期开展**

1. 使验收过程具有客观性
2. 通过确认每个可交付成果来**提高最终产品、服务或成果获得验收的可能性**

主要有干系人从控制质量过程输出的核实的可交付成果，确认这些可交付成果已经圆满完成并通过正式验收。
1. 确认范围的步骤：确认项目范围贯穿项目的始终。**确认范围前需要先进行质量控制工作**。**确认范围关注可交付成果的验收，质量控制关注满足质量要求。**
   1. 确定需要确认范围的时间
   2. 识别范围确认需要哪些投入
   3. 确定范围正式接受的标准和要素
   4. 确定范围确认
2. **需要检查的问题**
   1. 可交付成果是否确定的、可确认的
   2. 是否有明确的里程碑
   3. 是否有明确的质量标准：可交付成果和其标准之间是否有明确联系。
   4. 审核和承诺是否有清晰的表达
   5. 项目范围是否覆盖了需要完成的产品或服务的所有活动
   6. 项目范围的风险是否太高
3. 干系人关注的不同
   1. 管理层：项目进度、资金和资源的影响，是否超过承受范围、投入产出是否合理。
   2. 客户：可交付成果是否满足产品或服务
   3. 项目管理人：可交付成果能否完成

## 15 项目成本管理
是为了项目在批准的预算内完成，对成本进行规划、估算、预算、融资、筹资、管理和控制的过程。

**项目成本失控的原因****（人技料法环）**
1. 人:认识不足：对工程项目认识不足
   1. 对难度估计不足
   2. 工程项目的规模不合理
   3. 人员缺乏成本意识
   4. 对项目成本的使用缺乏责任感
2. 技:技术限制：技术的制约
   1. 对项目相关信息了解不深，项目规划设计不够完善，不能满足成本估算的需求
   2. 采用的项目成本估算方法不恰当
   3. 项目成本计算的数据不准确或有漏项
   4. 设计者未对设计方案进行优化
   5. 物资或设备价格的上涨
   6. 项目规划和设计方面的变更引起相关成本的增加
   7. 对工程实施中可能遇见的风险估计不足
3. 料:管理不当：需求管理不当
4. 法:方法有误：方法问题
   1. 缺乏用于项目投资控制所需要的方法
   2. 缺乏系统的成本控制程序和明确的具体要求
   3. 缺乏科学、严格、明确且完整的成本控制方法和工作制度
   4. 缺乏对计算机辅助投资控制程序的利
   5. 缺乏对计划值与实际值进行动态的比较分析
5. 环:制度不全：组织制度不健全
   1. 制度不完善
   2. 责任不落实
   3. 没有明确的投资分工

术语和定义：
1. 项目全过程所耗用的各种成本的总和为项目成本。
2. 产品的全生命周期成本：在获得阶段、运营与维护及生命周期结束时所发生的全部成本

**成本的类型**：
1. 可变成本：随生产量、工作量或时间而变的成本
2. 固定成本：不随生产量、工作量或时间而变的成本
3. 直接成本：直接归属于当前项目工作的成本
4. 间接成本：多个项目共同承担的成本
5. 机会成本：利用一定时间或资源用于交付一个产品或服务，而失去利用这些资源生产其他产品或服务
6. 沉没成本：指由千过去的决策已经发生的，而不能由现在或将来的任何决策改变的成本

应急储备和管理储备
和进度成本相似

成本基准：是经批准的按时间安排的成本支出计划，被用于度量和监督项目的实际执行成本。

项目成本管理过程：
1. 规划成本管理
2. 估算成本
3. 制定预算
4. 控制成本

### 制定预算
是汇总所有单个活动或工作包的估算成本，**建立一个经批准的成本基准**的过程。

3. 数据分析

## 12.1 质量管理

1. ISO定义的质量是反映实体满足主体明确和隐含需求的能力的特性总和
1. 国标定义的质量是一组固有特性满足要求的程度

广义的质量还包括工作质量。

质量与等级的区别
- 质量是一系列内在特性满足要求的程度
- 等级是对用途相同但技术特性不同的可交付成果的级别分类  
   例如：低等级、高质量也能被认可。高等级、低质量不会被认可。

预防胜于检查。

术语：
1. 预防：保证过程中不出现错误
2. 检查：保证错误不落到客户手中
3. 公差：结果可接受的范围
4. 控制界限：在统计学意义上稳定的过程或过程绩效的普通偏差的边界

项目质量：包括工作质量和产品质量。是顺应顾客要求进行的，其意图反应在合同中，**项目合同通常是进行项目质量管理的依据**。

### 质量管理：
质量管理 (Quality Management) 是指确定质量**方针、目标和职责**，并通过质量体系中的质量规划、质量保证、质量控制以及质量改进来使其实现所有管理职能的全部活动。
- 质量方针：由组织的最高管理者正式发布的该组织总的质量宗旨和方向。
- 质量目标：是指“在质量方面所追求的目的＂，它是落实质量方针的具体要求，它从属千质量方针。
  
按有效性递增排列的五种质量管理水平：
1. **让客户发现缺陷**，可能导致召回、商誉受损和返工成本
2. **控制质量过程中包括先检测和纠正缺陷**，再将可交付成果发送给客户。该过程会带来相关成本，主要是评估成本和内部失败成本。
3. **通过质量保证检查并纠正过程本身**
4. **将质量融入项目和产品的规划和设计中**
5. 再整个组织内**创建一种关注并致力于实现过程和产品质量的文化**

质量标准体系：
1. GB/T 19000系列标准：国标
2. **全面质量管理（TQM）**：全员、全过程、全组织的品质管理
   1. 全员参加的质量管理
   2. 全过程的质量管理
   3. 全面方法的质量管理
   4. 全面结果的质量管理

管理新实践：
**休哈特提出**由**戴明完善的**“计划-实施-检查-行动”（**PDCA**）循环是质量改进的基础。 全面质量管理TQM、六西格玛和精益六西格玛也可以提高质量

- 规划质量管理过程关注**工作需要达到的质晕**
- 管理质量则关注**管理整个项目期间的质量**，以规划质量管理的规划作为标准管理质量
- 控制质量**工作成果与质量要求的比较**，确保结果可接受


敏捷与适应方法
敏捷或适应型方法要求多个质量与审核步骤贯穿整个项目。

      1. 生理需求：员工宿舍、工作餐、工作服、班车、工资、补贴、奖金
      2. 安全需求：五险一金
      3. 社会交往的需求：员工活动、聚会、比赛、俱乐部
      4. 受尊重的需求：荣誉性奖励、形象、地位提升、颁发奖章、称为导师

      2. 激励因素：成就、
      - 不能忽视保健因素，也没有必要过分地改善保健因素。
   3. 麦格雷戈X理论和Y理论
      在团队的开始阶段，需要运用X理论去指导和管理。当进入执行阶段后，可以运用Y理论授权团队完成所负责分工作。
      1. X理论
         1. 人性本恶，只要有可能就会逃避工作
         2. 人以自我为中心，漠视组织要求
         3. 人缺乏进取心，逃避责任，甘愿听从指挥，安千现状，没有创造性
         4. 人们通常容易受骗，易受人煽动
         5. 人们天生反对改革
         6. 工作动机就是为了获得经济报酬。
      2. Y理论：与X相反
   4. 期望理论：期望概率并不是越大越好，也不是越小越好。当期望概率远高于实际概率可能会产生挫折，而太低又会减小某一目标的激发力量。
      激发力量 = 目标效价 X 期望值

## 20 项目风险管理 1
项目风险包括对项目目标的威胁和促进项目目标的机会。未知的风险无法管理。风险的不确定性随项目的进展而逐渐降低。

### 项目风险概述
1. 单个项目风险：进度风险，成本风险
2. 整体项目风险：单个风险和不确定性的其他来源联合导致的风险

### **风险的属性**
1. 风险事件的随机性
2. 风险的相对性：相同风险对不同的主体有不同的影响
   1. 收益的大小：收益越大，人们愿意承担的风险也就越大
   2. 投入的大小：投入越多，成功的希望越大，愿意承担的风险越小
   3. 项目活动主题的地位和拥有的资源：资源越多，风险承受能力越大
3. 风险的可变性：
   1. 风险的性质变化
   2. 风险的后果变化
   3. 出现新风险

### 风险的分类
1. 按照风险后果划分：纯粹风险和投机风险会互相转换。一般涉及风险的各个方面都要蒙受损失，无一幸免
   1. 纯粹风险：不能带来机会，无获得利益的可能。
   2. 投机风险：可能带来机会获得利益，有隐含威胁、照成损失的风险。
2. 按风险来源划分
   1. 自然风险
   2. 人为风险
3. 按风险是否可管理划分：能否管理取决于是否可以消除以及活动主体的管理水平。有些不可管理风险可以变为可管理风险。
   1. 可管理风险
   2. 不可管理风险
4. 按照风险的影响范围划分：要特别注意总体风险
   1. 局部风险
   2. 整体风险
5. 按风险后果的承担者划分
6. **按照风险的可预测性划分**
   1. 已知风险：经常发生的，而且其后果亦可预见的风险。已知风险发生概率高，但一般后果轻微，不严重
   2. 可预测风险：可以预见其发生，但不可预见其后果的风险。后果有时可能相当严重。
   3. 不可预测风险有可能发生，但其发生的可能性即使最有经验的人亦不能预见的风险

### 风险成本及其负担
1. 有形成本
   1. 有形成本
   2. 间接成本
2. 无形成本
   1. 风险损失减少了机会
   2. 风险阻碍了生产率提高
   3. 风险造成资源分配不当
3. 预防与控制
4. 风险成本的担负：不止项目主体承担，项目相关方也承担。

非事件类风险
1. 变异性风险：风险改变状况
2. 模糊性风险：风险内容和结果不确定

### 规划风险管理
是定义如何实施项目风险管理活动的过程.

作用是确保风险管理的水平、方法和可见度与项目风险程度相匹配，与对组织和其他干系人的重要程度相匹配。

立项阶段就应开始，并在项目早期完成。项目生命周期的后期，可能有必要重新开展本过程

#### 输入

1. 项目章程
2. 项目管理计划
3. 项目文件

### 规划风险应对
是为了应对项目风险，而制定可选方案、选择应对策略并商定应对行动的过程

作用是：
1. 制定应对整体项目风险和单个项目风险的适当方法
2. 分配资源

整个项目期间开展。

**风险应对方案应该与风险的重要性相匹配，并且能够经济有效地应对挑战，同时在当前项目背景下现实可行，获得全体干系人的同意，并由一名责任人具体负责**

1. 专家判断
2. 数据收集

4. **威胁应对策略**
   1. 上报：威胁一旦上报，就不再由项目团队做进一步监督，虽然仍可出现在风险登记册中供参考。
   2. 规避：有些风险可通过澄清需求、获取信息、改善沟通或取得专有技能来加以规避。具体规避措施包括
      1. 消除威胁的原因
      2. 延长进度计划
      3. 改变项目策略
      4. 缩小范围
   3. 转移：转移涉及将应对威胁的责任转移给第三方，让第三方管理风险并承担威胁发生的影响。
   4. 减轻：冗余，来减轻风险。
   5. 接受：此策略可用于低优先级威胁，也可用于无法以任何其他方式经济有效地应对的威胁。
      1. 主动：建立应急储备，包括预留时间、资金或资源以应对出现的威胁。
      2. 被动：定期对威胁进行审查，确保其并未发生重大改变。
5. 机会应对策略
   1. 上报
   2. 开拓：如果组织想确保把握住高优先级的机会，就可以选择开拓策略。将特定机会的出现概率提高到 100%。
   3. 分享：建立合伙关系、合作团队、特殊公司和合资企业分享机会。
   4. 提高：提高策略用千提高机会出现的概率和影响。提高概率和收益。
   5. 接受
6. 应急应对策略：特定事件发生时才采用的应对措施。采用此技术制订的风险应对计划通常称为应急计划，其中包括已识别的，用于启动计划的触发事件
7. 整体项目风险应对策略
   1. 规避
   2. 开拓
   3. 转移或分享
   4. 减轻或提高
   5. 接受
8. 分析工具
   1. 备选方案分析
   2. 成本收益分析
9.  决策

### 管理干系人参与
执行过程

是通过与干系人沟通协作，以满足其需求与期望，处理问题，并促进干系人合理参与的过程。

作用是，尽可能提高干系人的支持度，并降低干系人的抵制程度。

整个项目期间开展。

**需要开展多项活动**
1. 适当引导干系人参与，以便获取、确认或维持他们对项目的承诺
2. 通过谈判和沟通的方式管理干系人期望
3. 处理与干系人管理有关的任何风险或潜在关注点
4. 澄清和解决已识别的问题


#### 输入

1. 项目管理计划
2. 项目文件
3. 事业环境因素
4. 组织过程资产

#### 技术

1. 专家判断
2. 沟通技能
3. 人际关系与团队技能
   1. 冲突管理
   2. 文化意识
   3. 谈判
   4. 观察和交谈
   5. 政策意识
4. 基本规则：根据团队章程中定义的基本规则，明确项目团队成员与其他干系人引导干系人参与的行为。
5. 会议

### 监督干系人参与
是监督项目干系人的关系，并通过修订参与策略和计划来引导干系人合理参与项目。

作用是随着项目进展和环境变化，维持或提升干系人参与活动的效率和效果。

整个项目期间开展。


#### 输入

1. 项目管理计划
2. 项目文件
3. 工作绩效数据
4. 事业环境因素
5. 组织过程资产

#### 技术

1. 数据分析
   1. 备选方案分析
   2. 根本原因分析
   3. 干系人分析
2. 决策
   1. 多标准决策分析
   2. 投票
3. 数据表现：干系人参与评估矩阵。
4. 沟通技能
   1. 反馈：确保给干系人的信息被接受和理解
   2. 演示：为干系人提供清晰的信息
5. 人际关系与团队技能
   1. 积极倾听
   2. 文化意识
   3. 领导力
   4. 人际交往
   5. 政策意识
6. 会议

#### 输出

1. 工作绩效信息
2. 变更请求
3. 项目管理文件（更新）
4. 项目文件（更新）
