
## 项目整合管理2
### 监控项目工作
作用是监控项目工作是跟踪、审查和报告整体项目进展。**在整个项目期间开展**
#### 输入: 
1. 项目管理计划
2. 项目文件
3. 工作绩效信息
4. 协议
5. 事业环境因素
6. 组织过程资产
#### 技术
1. 专家判断
2. 数据分析（备选方案分析、成本效益分析、挣值分析、根本原因分析、趋势分析、偏差分析）
3. 决策（一致同意、大多数原则、相对多数原则）
4. 会议
#### 输出: 
1. 工作绩效报告：一般包括**状态报告**和**进展报告**。
2. 变更请求：
  1. 纠正措施
  2. 预防措施
  3. 缺陷补救
3. 项目管理计划（更新）
4. 项目文件（更新）

### 实施整体变更过程
作用是是审查所有变更请求、批准变更、管理对可交付成果、项目文件和项目管理计划的变更。**整个项目期间开展。项目经理承担最终责任**。任何时间任何干系人都可以提出变更。**基准确立前变更无需受控**。变更可以口头提出，但是要书面记录，并纳入变更管理和配置配置系统（包括被拒绝）。
#### 输入: 
1. 项目管理计划
2. 项目文件
3. 工作绩效报告（资源可用情况、进度和成本数据、挣值报告、燃烧图或燃尽图）
4. 变更请求（**变更可能影响基准**，通常由项目经理做决策。对于会影响基准的变更，由CCB和客户或发起人审批。）
5. 事业环境因素
6. 组织过程资产
#### 技术：
1. 专家判断
2. 变更控制工具
变更控制可以使用手动或信息化工具。
配置控制重点关注可交付成果及各个过程的技术规范。
变更控制主要是识别、记录、批准或否决对项目文件、可交付成果或基准的变更。
  1. 配置管理工具
    1. 识别配置项
    2. 记录并报告配置项状态
    3. 进行配置项核实与审计
  2. 变更控制工具
    1. 识别变更  
    2. 记录变更
    3. 跟踪变更
3. 数据分析(备选方案分析，成本效益分析)
4. 决策（投票、独裁、多标准）
#### 输出：
1. 批准的变更请求：批准、推迟或否决
2. 项目管理计划（更新）
3. 项目文件（更新）：变更日志

### 结束项目或阶段
作用是终结项目、阶段或合同的所有活动的过程。**仅仅开展一次**。主要作用
1. 存档项目或阶段信息，完成计划工作
2. 释放组织团队资源

结束项目或阶段时需要执行的活动：
1. 为达到项目或阶段结束所需要的必要活动
2. 为达到下一阶段开始所需要的必要活动
3. 当前阶段或项目的验收
4. 干系人的满意程度

如果提前结束，需要制定程序调查和记录结束的原因。项目经理应该引导**所有适合的干系人**参与结束项目或阶段的工作
#### 输入
1. 项目章程
2. 项目管理计划
3. 项目文件
4. 验收的可交付成果
5. 立项管理文件
6. 协议
7. 采购文档
8. 组织过程资产
#### 技术
1. 专家判断
2. 数据分析
  1. 文件分析：评估现有文件总结经验教训，以改进未来项目和组织资产
  2. 回归分析：不同项目变量间的相互关系，以提高未来绩效
  3. 趋势分析
  4. 偏差分析
3. 会议
#### 输出
1. 项目文件（更新）
2. 最终产品/服务/成果
3. 项目最终报告
  1. **项目或阶段概述**
  2. **范围目标**、范围评估标准
  3. **质量目标**、项目和产品质量的评估标准、相关核实信息和实际里程碑交付日期以及偏差原因
  4. **成本目标**，包括可接受的成本区间、实际成本，产生任何偏差的原因
  5. **最终产品**、服务或成果的确认信息的总结
  6. **进度计划目标**，包括成果是否实现项目预期效益：如果未能实现效益，则效益实现程度并预计未来实现情况
  7. **需求完成目标**关于最终产品、服务或成果如何满足业务需求的概述：如果未能满足，则何时能得到满足
  8. **风险或问题**及其解决情况
4. 组织过程资产(更新)

## 12 项目范围管理

![Alt text](../assets/img/pm/%E9%A1%B9%E7%9B%AE%E8%8C%83%E5%9B%B4%E7%AE%A1%E7%90%86%E8%BF%87%E7%A8%8B.png)

做且只做所需的全部工作。
管理基础：
1. 产品范围：产品范围是根据**产品需求**来衡量的。
2. 项目范围：项目范围是根据**项目管理计划**来衡量的

范围管理子过程
1. 规划范围管理
2. 收集需求
3. 定义范围
4. 创建WBS
5. 确认范围
6. 控制范围

### 裁剪考虑因素
1. 知识和需求管理
2. 确认和控制
3. 开发方法
4. 需求的稳定性
5. 治理

### 敏捷和适应的方法
- 在预测型项目中，经过批准的项目**范围说明书、工作分解结构（WBS）和相应的WBS词典构成项目基准**

### 规划范围管理
在整个项目中为管理项目管范围做指南和规划。**开展一次或在预定节点开展**
#### 输入
1. 项目章程
2. 项目管理计划：**质量管理计划**、项目生命周期描述、开发方法
3. 事业环境因素
4. 组织过程资产
#### 技术
1. 专家判断
2. 数据分析
3. 会议
#### 输出
1. 范围管理计划：可以是正式或非正式的、可以是详细的或者高度概括的
  1. 制定项目范围说明书
  2. 创建WBS
  3. 如何审批和维护范围基准
  4. 正式验收已完成的项目可交付成果
2. 需求管理计划

### 收集需求
作用是为定义产品范围和项目范围奠定基础。**开展一次或预定节点开展。**
#### 输入
1. 立项管理文件
2. 项目章程
3. 项目管理计划
4. 项目文件（假设日志、干系人登记册、经验教训登记册）
5. 协议
6. 事业环境因素
7. 组织过程资产
#### 技术
1. 专家判断
2. 数据收集（头脑风暴、访谈、焦点小组、问卷调查、标杆对照）
3. 数据分析（数据分析技术是文件分析）
4. 决策（投票、独裁、多标准）
5. 数据表现（亲和图、思维导图）
6. 人际关系和团队技能（**名义小组**、观察和交谈、引导）
7. 系统交互图
8. **原型法**
#### 输出
1. 需求文件：只有明确的、可跟踪的、完整的、相互协调的、且主要干系人认可的需求才能作为基准
   需求类别
   1. 业务需求：整个组织高层需求，解决业务问题或抓住业务机会，以及实施项目的原因
   2. 干系人需求
   3. 解决方案需求（功能需求，非功能需求）
   4. 过度和就绪需求
   5. 项目需求：项目需要满足的行动、过程或其他条件，交付日期之类
   6. 质量需求
2. **需求跟踪矩阵**：每个需求于业务目标和项目目标联系起来，确保每个需求都有业务价值。
   内容包括
   1. 业务需要、机会、目的和目标
   2. 项目目标
   3. 项目范围和WBS可交付成果
   4. 产品设计
   5. 产品开发
   6. **测试策略和测试场景**
   7. 高层及需求到详细需求

  典型特征：需要有测试用例
  1. 唯一标识
  2. 文字描述
  3. 收集需求的理由
  4. 所有者
  5. 来源
  6. 优先级别
  7. 版本
  8. 当前状态和状态日期
  9. 稳定性、复杂性、验收标准

### 定义范围
作用是描述产品、服务或成果的边界和验收标准。**整个项目期间多次反复开展**
#### 输入
1. 项目章程
2. 范围管理计划
3. 项目文件（假设日志、需求文件、风险登记册）
4. 组织过程资产
5. 事业环境因素
#### 技术
1. 专家判断
2. 数据分析
3. 决策
4. 人际关系和团队技能
5. 产品分析：定义产品和服务，描述产品的用用途、特征和其他方面
#### 输出
1. **项目范围说明书**
   对项目范围、主要可交付成果、假设条件和制约因素的描述。
   1. 产品范围描述
   2. 可交付成果：任意过程、阶段或项目产出的任何独特且可核实产品、成果或服务能力，也包括辅助成果
   3. 验收标准
   4. 项目的除外责任
2. 更新项目文件（假设日志、需求文件、风险登记册、干系人登记册）

### 创建WBS
作用是是吧项目可交付成果和项目工作分解成较小的、更易管理的组件的过程。
WBS最底层的组成部分称为**工作包**。工作包是可交付成果，不是活动本身。
**仅开展一次或仅在项目的预定义点开展**
#### 输入
1. 项目管理计划
2. 项目文件（需求文件，项目范围说明书）
3. 事业环境因素
4. 组织过程资产
#### 技术
1. 专家判断
2. **分解**：将项范围和项目可交付成果逐步划分为更小、更便于管理的组成部分的技术。工作分解得越细致，对工作的规划、管理和控制就越有力。但是，过细的分解会造成管理努力的无效耗费资源使用效率低下、工作实施效率降低，同时造成 WBS 各层级的数据汇总困难。
   1. 分解活动
      1. 识别和分析可交付成果和相关工作（看有什么需要分解）
      2. 确定wbs结构和编排方法（确定分解方法）
      3. 自上而下逐层细化分解（开发分解）
      4. 为WBS组成部分制定（编码标识分解内容）
      5. 核实可交付成果分解程度是否恰当（检查分解内容）
    2. WBS结构：
       1. 项目生命周期作为第二层，，把产品和项目可交付成果放在第三层
       2. 可交付成果作为第二层
       3. 纳入由项目团队意外的组织的各种较低层次组件（外包工作）
    3. 注意事项
       1. WBS 必须是面向可交付成果的
       2. WBS必须符合项目的范围：100%原则，在WBS中所有下级元素的和必须代表上一级元素
       3. WBS的底层应该支持计划和控制，**WBS是项目管理计划和项目范围之间的桥梁**
       4. WBS的元素必须有人负责，且只有一个人负责。
       5. WBS应该控制在4~6层。一个工作单元只能从属于某个上层单元，避免交叉从属。
       6. WBS应包括项目管理工作和外包工作
       7. WBS的编制需要**所有（主要）**项目干系人参与
       8. WBS并非一成不变的。完成之后也可修改。
#### 输出
1. 范围基准：是项目管理计划的组成部分
   1. 项目范围说明书
   2. WBS
   3. 工作包：WBS 的最低层是带有独特标识号的工作包。是已经确定的工作。**控制账号**是一个控制点，包含**两个以上的工作包**，一个包只能属于一个控制账号。
   4. 规划包：规划包是一种低千控制账户而高千工作包的工作分解结构组件。是待确定的工作。**控制账号**可以有**一个或多个规划包**。
   5. WBS字典：
2. 项目文件（更新）：假设日志、需求文件

### 确认范围
作用是正式验收已完成的项目可交付成果的过程。**在整个项目期间定期开展**
1. 使验收过程具有客观性
2. 通过确认每个可交付成果来**提高最终产品、服务或成果获得验收的可能性**

主要有干系人从控制质量过程输出的核实的可交付成果，确认这些可交付成果已经圆满完成并通过正式验收。
1. 确认范围的步骤：确认项目范围贯穿项目的始终。**确认范围前需要先进行质量控制工作**。**确认范围关注可交付成果的验收，质量控制关注满足质量要求。**
   1. 确定需要确认范围的时间
   2. 识别范围确认需要哪些投入
   3. 确定范围正式接受的标准和要素
   4. 确定范围确认
2. **需要检查的问题**
   1. 可交付成果是否确定的、可确认的
   2. 是否有明确的里程碑
   3. 是否有明确的质量标准：可交付成果和其标准之间是否有明确联系。
   4. 审核和承诺是否有清晰的表达
   5. 项目范围是否覆盖了需要完成的产品或服务的所有活动
   6. 项目范围的风险是否太高
3. 干系人关注的不同
   1. 管理层：项目进度、资金和资源的影响，是否超过承受范围、投入产出是否合理。
   2. 客户：可交付成果是否满足产品或服务
   3. 项目管理人：可交付成果能否完成
   4. 项目团队成员：工作是否有时间做，是否有多项冲突的工作，是否能够完成
#### 输入:
1. 项目管理计划：范围管理计划、需求管理计划、范围基准
2. 项目文件：需求文件、需求跟踪矩阵、质量报告、经验教训登记册
3. 工作绩效数据
4. 核实的可交付成果
#### 技术:
1. 检查：测量、审查、巡检
2. 决策
#### 输出:
1. 验收的可交付成果
2. 变更请求
3. 工作绩效信息
4. 项目文件（更新）：

### 控制范围
作用是监督项目和产品的状态，管理项目基准变更的过程。整个项目期间对项目范围基准的维护。**在整个项目期间开展**

#### 输入
1. 项目管理计划
2. 项目文件
3. 工作绩效
4. 组织过程资产
#### 技术
1. 数据分析：偏差分析，趋势分析
#### 输出
1. 工作绩效信息
2. 变更请求
3. 项目管理计划（更新）
4. 项目文件（更新）
